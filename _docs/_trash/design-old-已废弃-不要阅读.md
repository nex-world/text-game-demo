
这是一个极简的文字游戏项目。

#### 开发要求

- 尽可能极致简化，避免任何不必要的复杂性
- 禁止使用 enum, 这是一种糟糕的做法，应该直接使用联合类型来代替。
- 不允许直接在 package.json 中编辑依赖项，必须使用 pnpm 来管理。


#### 技术栈

- TypeScript, Vite
- React, tsx
- Zustand
- UI: Tailwind CSS + shadcn
- 存档管理: idb
- 校验: Zod

#### 游戏界面

- 极简的界面，不需要任何美观层面的考虑
- 从上到下布局
  - 状态区: 展示玩家当前状态
    - 时间, 地点, 角色属性
  - 历史事件区: 分为 3 个 tab
    - 综合: 展示所有事件
    - 个人经历: 仅列出玩家角色亲身经历的事件
    - 见闻: 仅列出玩家了解到的其他角色或世界性的事件
  - 当下信息区: 展示与当前行动紧密关联的信息
    - 比如正在战斗的敌人的数值信息
    - 比如正在交易的角色的财物信息和自己的财物信息
    - 比如打开自己的背包时展示自己的物品信息
    - 比如查看自己详细数值面板时展示相关信息
  - 操作区: 列出当下可执行的操作，可以有多级菜单

#### 游戏核心机制

##### 空间

- 这个游戏的空间是一个图状结构
- 图中每个节点是一个「地点」
  - 可能是一个具体的建筑、小型村庄之类
  - 地点中可能有若干游戏对象（角色、建筑、设施、道具）等
- 地点之间可能由「路径」相连
  - 每条路径有一个长度数值，影响通行时间
  - 相同两个地点之间，可能有不同的路径，比如陆路、水路、空路、传送门等
  - 不同路径可能有不同的通行条件
  - 不同路径可能触发不同随机事件
- 一整块相关联的「地点」构成一个「地区」
  - 地区之间的交通也是由地点之间的路径提供的
  - 可能是一个城市、一片森林、一座山脉等等
- 若干「地区」构成一片「地域」
  - 可能是一个国家、一片大陆等等
- 若干片「地域」构成一个「世界」
  - 通常是一个星球

##### 游戏对象

- 类型
  - 可以到处移动且有一定自主性的角色或生物，比如玩家、NPC、牲畜、野兽、怪物之类
  - 自主性相对弱的角色或生物，比如宠物、坐骑、召唤物等
  - 可收纳和携带的物品，比如战斗装备、数值消耗品、便携交通工具、功能道具等
  - 固定在特定地点或路径使用的固定设施（中小型）、公共交通工具、自然资源（植物、农作物、矿物等）之类
  - 与地点直接关联的土地、建筑、大型固定设施之类

##### 时间进程

- 玩家的每个行动都会消耗若干个时间单位
  - 1个时间单位 相当于 10分钟
  - 1小时 换算为 6个时间单位
  - 1时段 = 2时辰 = 4小时 = 24个时间单位
  - 1天 = 6时段 = 12时辰 = 24小时 = 144个时间单位
    - 0~4时  凌晨
    - 4~8时  早晨
    - 8~12时  上午
    - 12~16时  下午
    - 16~20时  傍晚
    - 20~24时  晚上
  - 1月 = 30天
  - 1个季 = 3个月
  - 1年 = 4个季 = 12个月
- 玩家行动的不同时间跨度
  - n时间单位 : 非常具体或短暂的事情，比如具体战斗动作、对话、操作道具和设施、近距离移动等
  - n小时 : 较为具体或稍微耗时的事情，比如采集、制作简单道具、休息、前往某些地点
  - n天 : 相对概况或有点耗时的事情，比如建造中型设施、长途交通、等待某个事件的发生等
  - n个月 : 较为模糊或相当耗时的事情，比如炼制高级道具、建造高级设施、建造大型建筑等
- 外部事件的更新与玩家消耗的时间单位以及玩家的能力范围有关
  - 玩家所处现场的事件 : 每时间单位更新
    - 影响当前互动的人物、道具、设施等对象的具体数值状态
  - 玩家所在地点的事件 : 每时段更新
    - 影响角色和生物的作息，比如商人的营业时间、野兽的行踪等
    - 触发与玩家紧密关联的随机事件之类
  - 地区事件 : 每天特定时段更新
    - 比如听闻一些重要 npc 的行动、得知一些重要设施的运转状况等
  - 地域事件和世界事件 : 每月更新
    - 影响整个地域或世界的重大事件，比如战争、灾难、政治变动等
- 大时间跨度的事件更新不是立即触发的，而是先 schedule 起来，等到时间点到了再触发

- 事件描述
- 事件级别（地点/地区/地域/世界）【用于在某个级别触发事件的】

##### 核心循环

- 读取数值状态
- 数值驱动事件
- 玩家行动
- 状态更新

首先，读取数值状态，获得玩家当前可以采取的行动列表，然后，玩家选择一个行动。然后，计算这个行动对数值的影响来更新数值状态，并且计算行动过去的时间，然后根据时间更新事件，再根据这些事件的影响更新数值状态。接着，进入下一个循环。

游戏更新循环：
- 每个时间单位更新一次
- 更新当前时间单位发生的事件（是否有新的事件触发）
- 检查当前时间单位正在发生的所有事件，是否在当前时间单位结算（根据结算周期判断）
- 检查当前时间单位正在发生的所有事件，是否在当前时间单位结束（包括到事件自然结束或者被其他新发生的事件打断）
- 查看当前玩家动作，如果玩家当前没有正在执行的事件（包括之前执行的事件刚刚结束了，此时就空闲了；或者本来空闲的，因为新触发的事件强制占据了玩家的事件，就不空闲了）
- 如果玩家当前空闲，计算当前可以采取的行动列表，等待玩家做出行动选择
- 如果玩家有正在执行的事件，查看当前时间是否需要玩家操作，如果需要，出现操作选项等待玩家操作
- 玩家行动后，更新玩家正在执行的事件状态
- 进入下一个更新循环
